# pdfree 编辑内核架构（v2 方案）

2026-07-09 定稿。背景：v1 引擎是「内容流补丁器」——单指令级替换 + 字距补偿。
它保证了编辑点外像素不变，但结构上到头了：字数变多会重叠（补偿把后文钉死在原位）、
无样式概念（改字体/字重/颜色无从谈起）、Skia 单字形指令文件无法整词编辑。
v2 的目标是真正的 PDF 编辑内核：引擎是资产，web 只是壳。

## 分层

```
L0 对象层      lopdf + salvage.rs          （已有）xref/对象/流，容错加载
L1 内容层      walk.rs 解释器              （已有）操作符 + 图形状态跟踪
L2 文本模型    model.rs                    （新）★ 本次重构的核心
L3 编辑操作    edit.rs                     （新）模型上的语义操作
L4 行内排版    layout.rs                   （新）编辑后的局部重排
L5 字体服务    borrow / type3gen / ttf     （已有，升级为统一 FontService）
     验证闭环 harness 贯穿所有层（不变，判定区域从"编辑框"升级为"重排区域"）
```

## L2 文本模型（Phase A）

从 walk 的 Seg 流重建逻辑结构：

```
Glyph  单字形：unicode + 字节码 + 字体 + 精确定位（来自 Tm 链）+ 填充色
Run    同字体/字号/颜色/基线连续的字形串（Skia 的碎指令在这里合并成词句）
Line   基线聚类后的一行（跨 Run）
Block  几何邻接的行组（段落近似，供 L4 用）
```

- 每个 Glyph 记录**溯源**：来自哪条 op、TJ 第几个元素、字节区间 —— 模型到内容流可双向映射
- walk.rs 需要补录填充色（rg/g/k/sc/scn 状态）和逐字形位置（现在只有段级 bbox）
- extract JSON 改为输出 Line/Run 树，web 端按 Run 选择、按词点击
- **立刻解锁**：Skia 文件的整词编辑（匹配在模型层做，与指令碎片无关）

## L3 编辑操作（Phase B/C）

所有编辑表达为模型操作，实现方式统一为**行级重生成**：
不再给单条指令打补丁，而是删掉受影响行的全部原始 op（模型有溯源），
按编辑后的模型重新发射这一行的 op 序列。行外内容字节不动。

- `replace_text(range, new)`   —— 替换（Phase B）
- `set_style(range, {font/size/bold/italic/color})` —— 样式（Phase C）
  - 颜色 = 围绕 Run 发射 rg/RG，最简单，先做
  - 字重/字体 = 复用字体服务三级策略：文档内借字体（样式评分）→ 兜底家族选字重 → 报不可行
  - 字号 = Tf + 基线位置调整
- 插入/删除 = replace 的特例

## L4 行内排版（Phase B 同步 / D 扩展）

- **B：行内推挤**。编辑后重排本行字形：后续字形右移/左收，吃掉行尾空白。
  超出行右界 → 先诚实报 `NeedsReflow` 错误（不产出重叠！）
- **D：块内换行**。Block 近似段落，在块 bbox 内重新断行。
  永不做全局 reflow（那是排版软件，不是 PDF 编辑器）

## 验证闭环怎么跟着变

- 隔离裁判的判定区域 = 重生成的行/块 bbox（引擎上报），行外仍要求零 diff
- 新增裁判：样式编辑 → 采样像素颜色/字形对比；重排 → 文本顺序与内容不丢
- 新增语料测试项：变长替换（1→N 字）必须无重叠（渲染后字形 bbox 互斥检测）

## 迁移策略

Phase A 起 v1 补丁路径冻结不再加功能，修完 A+B 后删除 finish_swap/补偿机制。
每个 Phase 结束跑全量回归，通过率不得低于前一基线（当前 89.3%）。

Phase A  文本模型 + extract 改版 + web 按词选择      ← 下一步
Phase B  行级重生成 + 行内推挤（重叠问题在此消灭）
Phase C  样式操作（颜色 → 字重 → 字号 → 字体）
Phase D  块内换行 + 段落编辑

## 不变的铁律

MIT；净室（AGPL/GPL 实现永不参考，可读 pdfium/BSD 与 pdf.js/Apache）；
通过率是唯一 KPI；简单模块派弱模型 worktree，主 agent review 合入。
